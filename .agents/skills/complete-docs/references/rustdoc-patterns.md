# Rustdoc Patterns

## 模块文档

目标：先让读者知道“这层做什么、不做什么、从哪里读起”。

检查清单：
- 使用 `//!` 写模块级说明。
- 先写 1 到 3 句职责概览。
- 明确边界：输入、输出、状态、依赖或不负责的内容。
- 在 `# Intro Links` 与 `# 内部导航` 中二选一，不要同时写两个导航区块。
- 只有“给入口清单、没有明显阅读顺序”时使用 `# Intro Links`。
- 只有“需要指导阅读顺序或调用链”时使用 `# 内部导航`。
- 如果是门面模块，说明导出关系或推荐入口。

模板：

```rust
//! 一句话职责。
//!
//! 这层负责……，不负责……。
//!
//! # 内部导航
//! 建议阅读顺序：
//! 1. [`crate::foo::Bar`]：先看状态；
//! 2. [`crate::foo::Bar::run`]：再看主流程。
```

仅需入口列表时，改用：

```rust
//! # Intro Links
//! - [`crate::foo::Bar`]：核心类型；
//! - [`crate::foo::Bar::run`]：主入口。
```

## 类型文档

目标：解释类型为什么存在，而不是重复字段类型。

检查清单：
- 写清角色：状态容器、配置对象、领域事件、适配层 DTO（数据传输对象）等。
- 写清不变式、所有权边界、生命周期或使用前提。
- 写清与相邻类型的关系。
- 对公共类型优先补 rustdoc。

## 字段文档

目标：解释字段的业务语义、单位和可空语义。

检查清单：
- 说明字段含义，而不是只翻译名字。
- 如果有单位、时间戳口径、枚举来源、默认值来源，要写出来。
- 如果 `Option`、`Vec`、`HashMap` 有特殊语义，明确为空时代表什么。

## 函数 / 方法文档

目标：写清契约、边界与副作用。

检查清单：
- 写用途：这一步推进了什么状态或完成了什么动作。
- 写输入 / 输出：调用者需要提供什么，返回值代表什么。
- 写副作用：是否改状态、写日志、回写 portfolio（组合状态）、做 I/O。
- 写语义：幂等、顺序要求、时间戳口径、错误语义、忽略条件。
- 不把显而易见的赋值和 getter（只读访问器）硬写成长文。

模板：

```rust
/// 应用一条用户事件并回写组合状态。
///
/// 语义说明：
/// - 仅消费已经落地的成交事件；
/// - 基于事件 ID 去重，保证幂等；
/// - 优先使用事件内时间戳，缺失时回退到记录时间。
```

## Doctest

目标：让示例和真实 API 一起演进，而不是变成陈旧说明。

检查清单：
- 只给稳定、可复现、调用成本低的公共接口写 doctest。
- 示例尽量短小，聚焦一个行为。
- 结构迁移后同步更新模块路径和导入路径。
- 只有确实无法运行时才使用 `no_run` 或 `ignore`，并写清原因。
- 改文档后优先运行 `cargo test --doc`。

## 常见误区

- 不要用裸 `src/...` 路径冒充导航。
- 不要把“未来想做的设计”写成“当前行为”。
- 不要为了文档完整性补一堆空话。
- 不要把同一批入口同时写进 `# Intro Links` 和 `# 内部导航`。
- 不要只补模块文档而忽略结构迁移后的 README / 架构导航。
