# Project Docs Checklist

## README / 架构 / 计划文档

目标：让仓库级文档和当前代码结构、导出入口、验证状态保持一致。

检查清单：
- 先确认当前 owner（归属模块）和稳定入口，不根据旧路径延续说明。
- 结构迁移后同步更新 `README.md`、`docs/architecture/*.md`、`PLAN_*.md`。
- 导航优先写真实入口、模块根、重导出名，不写伪导航。
- 需要文件跳转时，使用真实可点击链接，不写裸路径段落充数。
- 计划文档中的“完成 / blocked / 待收口”口径必须和当前验证结果一致。

## 验收写法

结论优先写真实状态：
- 哪些命令通过；
- 哪些命令失败；
- 哪些问题已修；
- 哪些问题仍未收口。

推荐写法：
- `cargo fmt --all --check`：通过
- `cargo check`：通过
- `cargo test --doc`：通过
- `cargo test`：失败，原因是……

不要写法：
- “应该已经没问题了”
- “看起来完成了”
- “大体没问题，先标记完成”

## 结构迁移后的文档收口

当模块、目录、导出面或测试挂载发生变化时，同步检查：
- README 的模块导航是否还指向旧入口；
- 架构文档是否还写着已经删除的路径；
- doctest 示例是否仍能编译；
- 计划文档是否残留早期阻塞描述；
- 文档里的术语、类型名、函数名是否和当前代码一致。

## 文档改动边界

- 只修当前主题相关文档，不顺手扩到无关专题。
- 文档收口优先保证真实性，再考虑措辞精美。
- 如果代码语义未确认，明确写“基于当前实现推断”或保留待确认项。
