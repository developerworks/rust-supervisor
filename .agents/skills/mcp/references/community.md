# MCP Community

## 这层回答什么

Community（社区与治理层）回答的是：

- MCP 是谁在治理
- 谁能做决定
- 如何参与社区
- Working Groups / Interest Groups（工作组 / 兴趣组）如何运作

## 关键事实

- MCP 有正式治理模型，不只是松散开源协作。
- 官方治理页面明确区分多个角色层级，例如：
  - Lead Maintainers（最高最终决策层）
  - Core Maintainers（核心维护层）
  - Maintainers（具体领域维护者）
  - Contributors（贡献者）

## Working Groups 与 Interest Groups

### Interest Groups（兴趣组）

- 主要用于讨论问题、收集需求、形成问题空间
- 偏“找问题、聚共识”

### Working Groups（工作组）

- 主要用于推进具体交付物
- 常见交付物：
  - SEP
  - reference implementation（参考实现）
  - 持续维护的项目

## 什么时候用哪种组

- 还在探索问题、用例、需求：
  - 更适合 IG
- 已经有明确交付目标，要写提案或落地实现：
  - 更适合 WG

## 治理与提案的关系

- 社区不是 SEP 的替代品
- 社区提供讨论、组织、协调、赞助和推进机制
- SEP 提供正式的技术提案记录与决策载体

## 参与路径

典型路径通常是：

1. 跟进公开社区讨论
2. 加入相关 IG / WG
3. 贡献讨论、文档、实现、评审
4. 参与或发起 SEP

## 回答时的边界

- 不要把“Discord 讨论”直接等同于“正式标准决定”
- 不要把所有设计问题都说成必须先建 WG
- 也不要把正式治理简化成“谁先写代码谁说了算”

## 回答模板

- 如果用户问“我能不能直接提 SEP”：
  - 可以说理论上任何人都可以提交 SEP，但在相关工作组或社区里先打磨通常更有成功率。
