# Rust Supervisor(监督器) 宪章

## Core Principles(核心原则)

### I. Rust(编程语言) 模块所有权

所有生产行为必须放在边界清晰的 Rust(编程语言) 模块中。行为超出入口
连接代码后，`src/main.rs` 必须保持轻量，复用的运行时逻辑必须进入职责
明确的命名模块。对外 API(接口) 必须保持最小集合；项目禁止添加
compatibility exports(兼容导出)。理由：监督器运行时只有在所有权、可见性
和模块目的都能从代码布局中看清时，维护者才能可靠审查它。

### II. Supervision(监督) 语义必须成为契约

任何会启动、停止、重启、监控、取消、超时或恢复工作的功能，都必须在规格
和计划中说明这些语义，然后才能实现。该契约必须写明受监督单元、生命周期
状态、失败行为、关闭行为和调用者可见的结果。理由：监督代码属于控制平面，
隐含的生命周期行为会制造无法审查的失败模式。

### III. 行为变化必须先有测试

行为变化的任务列表必须先列出测试任务，然后才能列出实现任务。测试必须覆盖
被改变的生命周期、错误和边界行为；`cargo test` 必须通过后，变更才能算完成。
纯文档或纯模板变更可以使用静态校验，但计划或最终验证必须说明运行时测试不
适用的原因。理由：监督器行为常在边界处失败，所以测试必须跟行为一起规划。

### IV. 失败必须可观察

运行时失败路径必须返回结构化错误或诊断信息，并且必须指出受监督单元、
生命周期阶段和真实原因。预期失败禁止被 panic(恐慌)、被忽略的任务结果或
有损字符串消息掩盖。日志、tracing(结构化追踪) 或命令输出必须足以调试启动、
关闭、重启、超时和取消结果。理由：操作者和测试都需要精确信号来判断受监督
工作为什么失败。

### V. 增量必须小且可验证

每个功能必须先交付可以独立验证的最小切片。新依赖、后台工作者、异步运行时、
持久化层和跨模块抽象，都必须在计划中说明理由，然后才能实现。项目禁止
placeholder adapter(占位适配器)、fake integration(虚假集成) 和推测性的
compatibility layer(兼容层)。理由：监督器在失败压力下必须保持可理解；多余
抽象会遮住真实执行路径。

### VI. 中文写作和术语表达

规格、计划、任务、研究结论、数据模型、接口契约、快速开始、运行说明、审查
结论和最终汇报必须使用中文写作。句子必须符合汉语语法，即“主谓宾定状补，
定语放在主宾前，谓前是状，谓后是补”。项目禁止片段式语言，禁止非中文写作，
禁止使用生僻词，禁止使用方言。英文术语必须写成 `English(中文说明)`，禁止
只写单独的英文术语。代码标识符、文件路径、命令、包名和协议字段可以保持
原样，但周围说明必须使用中文。除非用户主动要求记住，工作过程禁止主动记忆。
理由：项目文档和沟通必须让中文读者直接理解，术语也必须在第一次出现时完成
解释。

## Rust(编程语言) 项目约束

项目使用 `Cargo.toml` 中声明的 Rust 2024(编程语言版本)。格式化和编译检查
必须使用 Cargo(构建工具) 工具链。运行时代码必须优先使用类型化数据结构、
明确的 `Result`(结果类型) 错误边界和窄可见性，禁止用临时字符串或全局可变
状态代替清晰模型。异步或线程监督必须通过类型或文档契约明确说明取消、join
(等待任务结束) 和关闭所有权。

源代码布局必须把功能代码放在 `src/` 下，把测试放在模块测试或 `tests/`
下，并且测试位置必须匹配被测行为的范围。文档变更必须和真实代码路径、命令
行为保持一致。

## 开发工作流

每个功能都必须从规格开始。规格必须写出独立用户故事，或者写出可独立测试的
技术切片。实现计划必须在 Phase 0(研究阶段) 前通过 Constitution Check(宪章
检查)，并且必须在 Phase 1(设计阶段) 后再次检查。任何违反宪章的地方，都
必须记录在 Complexity Tracking(复杂度跟踪) 中，并写明具体理由和被拒绝的
更简单方案。

任务必须按顺序排列。行为变化必须先写契约测试或回归测试，然后实现生产代码，
再更新文档，最后运行验证命令。工作必须保留工作区中与当前任务无关的用户变更。
当行为发生变化时，相关文档、模板或 quickstart(快速开始) 必须在同一轮变更
中同步。

## 写作和文档约束

Spec Kit(规格工具) 生成的 `spec.md`、`plan.md`、`tasks.md`、`research.md`、
`data-model.md`、`quickstart.md`、`contracts/` 和 `checklists/` 必须遵守
中文写作和术语表达原则。模板必须把该要求写进检查项，生成后的功能文档必须
能被人工检查。英文缩写或英文技术名第一次出现在正文中时，必须写成
`English(中文说明)`。后续若继续使用同一术语，仍然优先保留中文说明，避免
读者只能靠英文理解。

## Governance(治理)

本宪章高于冲突的本地习惯、模板和功能计划。修订宪章时，必须更新本文件，
必须包含 Sync Impact Report(同步影响报告)，并且必须在同一轮变更中把变更
规则传播到相关 Spec Kit(规格工具) 模板和运行时指导文件。

版本遵守 semantic versioning(语义化版本)。删除或重新定义原则时，必须增加
`MAJOR(主版本)`；新增原则或实质扩展治理要求时，必须增加 `MINOR(次版本)`；
只澄清文字且不改变义务时，必须增加 `PATCH(补丁版本)`。计划创建、任务生成
和最终验证期间都必须进行合规审查。任何不能满足原则的功能，都必须在实现前
写明例外理由。
