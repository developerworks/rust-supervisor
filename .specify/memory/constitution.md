# Rust Supervisor(监督器) 宪章

## 术语与适用范围(最小集合)

下列定义只用于解释本文件条文. 若下文出现未列出的词, 按通常的软件工程含义理解.

- **生产代码(production code)**: 会被编译进发布产物的 `src/` 内代码, 以及该代码在运行时读取并生效的配置schema(结构定义). 注释与文档不算生产代码, 但会改变运行时行为的配置示例若被工具默认加载则按生产代码相关处理.
- **监督行为(supervision behavior)**: 启动, 停止, 重启, 监控, 取消, 超时, 恢复或任何会改变受监督单元生命周期状态的自动化动作, 以及调用者能看见的错误, 事件, 退出码, 状态字段与日志语义.
- **受监督单元(supervised unit)**: 监督器创建或托管的进程, 任务, 句柄或等价执行上下文. 名称以规格与计划中的模型为准.
- **行为变化(behavior change)**: 任一满足以下任一条的变更:
  - 改变监督行为,
  - 改变面向调用者的命令行接口, 默认参数或稳定输出格式,
  - 改变预期失败时的错误类型, 错误字段或可观察诊断,
  - 改变取消, 关闭, join(等待任务结束) 或资源释放的时序或所有权约定.
  仅修改文档, 模板文字, 注释, 拼写, 且不改变上述可观察行为的, **不是**行为变化.
- **结构化错误(structured error)**: 具备稳定类型或稳定字段名的错误值; 仅有自由文本且无法让调用方可靠分支处理的, 不算结构化错误.
- **兼容导出(compatibility exports)**: 仅为兼容外部调用而存在的 `pub use` 重导出, 别名模块路径或薄封装层. 本仓库禁止新增此类导出; 是否撤销既有导出按单独迁移计划处理, 本文件不强制一步到位.

## Core Principles(核心原则)

### I. Rust(编程语言) 模块所有权

1. 所有生产行为必须落在命名清晰, 边界明确的 Rust(编程语言) 模块中.
2. `src/main.rs` 只能包含参数解析, 依赖拼装与进入模块 API(接口) 的连接代码. 可复用的运行时必须下沉到命名模块, 不得堆在入口文件中.
3. 对外可见的 `pub` API(接口) 必须保持最小集合. 禁止新增兼容导出.
4. **判定**: 审查者必须能仅从模块路径与可见性读出"谁拥有状态, 谁发起监督, 谁负责关闭".

**理由**: 监督器属于控制平面, 入口文件堆积逻辑会让所有权与审查边界失真.

### II. Supervision(监督) 语义必须成为契约

若变更会触碰监督行为, 在满足**实现合并**前必须同时满足:

1. `spec.md` 与 `plan.md` 写明受监督单元, 生命周期状态枚举或阶段表, 每种失败下监督器采取的动作, 关闭与取消语义, 调用者可见结果(含错误与状态).
2. 若某条监督能力本次不变更, 必须写 `N/A(不适用)` 并给出不适用原因, 避免隐含默认.
3. 契约必须与最终代码可对照: 代码中的状态名, 错误变体, CLI(命令行) 输出字段不得在无规格更新的情况下漂移.

**理由**: 隐含生命周期语义会产生不可审查的失败模式.

### III. 行为变化必须先有测试

对**行为变化**:

1. `tasks.md` 中必须先列出可执行的测试任务条目, 再列出实现生产代码的条目.
2. 测试必须覆盖被改变的生命周期路径, 错误路径与边界条件.
3. 合并前本仓库默认验证命令 `cargo test` 必须通过.

对**非行为变化**(例如仅文档, 模板或注释):

1. 允许不运行运行时测试, 但必须用**静态手段**完成验证, 并在 `plan.md` 或等效最终说明中写明所选静态手段与"为何不适用 `cargo test`".
2. **静态手段**至少覆盖以下之一: 人工 diff 审阅证明无可执行语义变化, 项目内脚本或 CI(持续集成) 检查, 对生成物或链接的一致性检查. 不得省略验证与说明.

**理由**: 边界行为是监督器故障高发区, 测试与验证必须与变更同属一批次规划.

### IV. 失败必须可观察

1. 所有预期失败路径必须向调用者返回结构化错误, 或在契约中约定的通道上输出可解析诊断; 诊断必须能指出受监督单元标识, 生命周期阶段与根因类别.
2. 禁止用 `panic!(恐慌)`, 丢弃 `Result`(结果类型), 忽略 `JoinHandle`(异步或线程句柄) 的结果, 或仅输出含糊一行字符串来掩盖预期失败.
3. 日志, tracing(结构化追踪) 或 CLI(命令行) 输出必须足以独立定位启动失败, 关闭失败, 重启失败, 超时与取消结果.

**理由**: 运维与自动化测试需要稳定, 可对比的信号.

### V. 增量必须小且可验证

1. 每个功能必须先交付可独立验证的最小切片; 切片必须在 `plan.md` 写清验收方式.
2. 在编码前, 若引入新 crate(库)依赖, 后台任务, 新的异步运行时设施, 持久化层或跨模块抽象, 必须在 `plan.md` 写明必要性, 并给出更简单方案被拒绝的原因.
3. 禁止 placeholder adapter(占位适配器), fake integration(虚假集成), 推测性 compatibility layer(兼容层) 进入合并默认路径.

**理由**: 失败压力下仍需可读的真实执行路径.

### VI. 中文写作和术语表达

1. 规格, 计划, 任务, 研究结论, 数据模型, 接口契约, quickstart(快速开始), 运行说明, 审查结论与最终汇报必须使用完整中文句子写作.
2. 句式必须符合现代汉语书面语法: 主语, 谓语, 宾语齐全; 定语位于中心语前; 状语位于谓语前; 补语位于谓语后. 禁止使用片段式列表冒充句子.
3. 禁止非中文主导叙述, 生僻词与方言化表达.
4. 英文或缩写首次出现在正文时, 必须写作 `English(中文说明)`. 后续同一概念优先继续带中文说明, 避免全英文堆叠.
5. 代码标识符, 文件路径, 命令, 包名与协议字段保持原文, 但其前后解释句仍须中文.
6. 除非用户明确要求记忆, 代理与工作流禁止写入持久化"记忆".

**理由**: 文档读者以中文为主, 术语必须在首次出现时自解释.

## Rust(编程语言) 项目约束

1. 语言版本以根目录 `Cargo.toml` 中 `edition` 与工具链声明为准; 本仓库目标为 Rust 2024(编程语言版本) 生态实践.
2. 格式化与静态检查必须通过 Cargo(构建工具) 官方或项目约定的子命令完成; 具体命令以 `README` 或 CI(持续集成) 为准, 本文件不锁死子命令名.
3. 运行时代码必须优先使用显式类型, 窄可见性与明确 `Result`(结果类型) 边界; 禁止以全局可变状态或临时字符串拼接顶替领域模型.
4. 凡使用异步或线程进行监督, 必须在类型签名, 状态机或文档契约中写明取消, join(等待任务结束) 与关闭的所有权与顺序.
5. 功能代码位于 `src/`; 行为测试位于对应模块测试或 `tests/`. 测试目录层级必须反映被测范围.
6. 任何文档变更必须自述与之对齐的代码路径, 命令与配置, 不得留下无法执行的描述.

## 开发工作流

1. 功能工作必须从 `spec.md` 起步; 规格必须落到"可独立交付的用户故事"或"可独立验证的技术切片"之一, 且边界清晰.
2. `plan.md` 在 Phase 0(研究阶段) 前必须完成首次 Constitution Check(宪章检查) 自检; Phase 1(设计阶段) 产出冻结前必须二次自检. 检查项以本文件与 `plan-template.md` 中的并列列表为准.
3. 任何明知偏离本宪章的决策必须写入 Complexity Tracking(复杂度跟踪) 小节, 包含: 偏离条目编号, 为何不可避免, 曾被否决的更简单方案摘要.
4. `tasks.md` 中任务必须按依赖顺序排列. 对行为变化, 默认顺序为: 测试任务 → 生产实现 → 文档与 quickstart(快速开始) → 项目规定的验证命令.
5. 进行本功能相关提交时, 不得擅自丢弃或覆盖工作区中与本功能任务无关的用户本地修改, 除非用户书面确认.
6. 若一次变更触达行为变化, 同一变更批次必须同步更新受影响规格, 模板或 quickstart(快速开始), 避免文档与代码口径分裂.

## 写作和文档约束

1. Spec Kit(规格工具) 生成的 `spec.md`, `plan.md`, `tasks.md`, `research.md`, `data-model.md`, `quickstart.md`, `contracts/` 与 `checklists/` 必须满足原则 VI.
2. 模板必须内嵌可勾选检查项, 以便人工审阅生成物.
3. 英文缩写或技术专名首次出现必须使用 `English(中文说明)`; 后续重复仍应带中文说明, 避免读者依赖隐式英语能力.

## Governance(治理)

1. 本宪章优先于冲突的本地惯例, 未升格的模板文字与未写明例外理由的功能计划条文.
2. 修订本文件时必须: 更新正文, 更新本文件顶部 `Sync Impact Report(同步影响报告)`, 在同一变更批次内传播到受影响的 Spec Kit(规格工具) 模板与运行时指导文档(例如 `AGENTS.md` 中指向宪章的段落).
3. 版本号遵守 semantic versioning(语义化版本):
   - 删除或重新定义原则: 递增 `MAJOR(主版本)`.
   - 新增原则或实质性扩展义务: 递增 `MINOR(次版本)`.
   - 仅澄清文字, 定义, 示例或排版且不改变义务集合: 递增 `PATCH(补丁版本)`.
4. 计划创建, 任务拆分与最终验证阶段必须进行合规审阅. 无法满足某项原则的交付必须在编码前取得明确的书面例外理由, 并落入 Complexity Tracking(复杂度跟踪).
