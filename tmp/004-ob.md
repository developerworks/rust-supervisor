本项目有没有做好全链路的可观测性准备, 比如要持久化所有事件到开源观测平台, 用于运行时的问题排查

结论是, 本项目已经有可观测性的核心骨架, 但还没有做好"全链路持久化到开源观测平台"的生产级准备.

已有部分主要在库内闭环. [manual/zh/observability.md](/Users/0x00/Documents/rust-supervisor-tools/rust-supervisor/manual/zh/observability.md:13) 明确了 `SupervisorEvent` 会派生结构化日志, tracing(追踪)事件, metrics(指标), audit(审计)事件, event journal(事件日志), test recorder(测试记录器). [src/observe/pipeline.rs](/Users/0x00/Documents/rust-supervisor-tools/rust-supervisor/src/observe/pipeline.rs:93) 也确实有 `ObservabilityPipeline`, 会把一次生命周期事实写入 journal(事件日志), metrics(指标), recorder(记录器), subscriber(订阅者). 这说明项目已经为排查问题建立了统一事件入口.

但是, 当前不是全链路持久化. [src/journal/ring.rs](/Users/0x00/Documents/rust-supervisor-tools/rust-supervisor/src/journal/ring.rs:1) 写得很清楚, `EventJournal` 是固定容量的近期事件缓冲区, 它会丢弃旧事件, 而且不负责 exporter(导出器). 这意味着它适合看最近状态, 不适合保证"所有事件都可追溯". [Cargo.toml](/Users/0x00/Documents/rust-supervisor-tools/rust-supervisor/Cargo.toml:37) 目前只有 `metrics` 和 `tracing`, 没有 `opentelemetry`, `otlp`, `prometheus exporter`, `loki`, `jaeger`, `tempo` 这些外部观测平台接入依赖.

更关键的是, [specs/001-create-supervisor-core/contracts/public-api.md](/Users/0x00/Documents/rust-supervisor-tools/rust-supervisor/specs/001-create-supervisor-core/contracts/public-api.md:288) 规定了每个生命周期事实要进入事件, 日志, 追踪, 指标, 审计, journal(事件日志)和测试记录器, 但同时也写明 core(核心库)不能绑定具体 exporter(导出器). 所以当前设计方向是对的, 但是缺少一个独立的导出和持久化切片.

如果你的目标是运行时问题排查, 后续至少需要补一个 `observability export pipeline(可观测性导出管线)`:

- 增加 `EventExporter(事件导出器)` 或 `ObservabilitySink(可观测性接收端)` 这种端口, core(核心库)只依赖抽象.
- 增加 durable queue(持久队列)或 spool(本地缓冲), 避免观测平台短暂不可用时丢事件.
- 增加 OpenTelemetry Collector(开放遥测采集器)接入, 通过 OTLP(开放遥测协议)输出 traces(追踪), logs(日志), metrics(指标).
- 明确 Prometheus(监控系统)只适合指标, 不适合保存所有事件.
- 对"所有事件"持久化, 更适合 Loki(日志系统), ClickHouse(列式数据库), PostgreSQL(关系数据库), 或对象存储中的 append-only event log(追加式事件日志).
- UI(用户界面)需要能按 `correlation_id(关联编号)`, `sequence(序号)`, `child_id(子进程编号)`, `generation(代际编号)` 跳到对应事件链路.

所以, 目前可以说"已经有可观测性基础", 不能说"已经做好全链路持久化准备". 004 系列实现后, 这个缺口会更明显, 因为生命周期守卫, 真实关闭流程, child slot(子任务槽位), generation fencing(代际隔离)都会产生更重要的运行时事件. 我建议把它单独立成后续规格, 例如 `005-observability-export-pipeline`, 不要塞进 core(核心库)里硬绑定某一个平台.
