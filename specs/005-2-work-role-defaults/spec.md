# Feature Specification(功能规格): 监督任务角色与默认策略

**Feature Branch(功能分支)**: `[005-2-work-role-defaults]`
**Created(创建日期)**: 2026-05-16
**Status(状态)**: Draft(草稿)
**Input(输入)**: 继承自主输入中对任务类别区分的要求: "`Permanent`(永久重启) 策略会在任务成功退出后也重启. 这对 `daemon`(常驻任务) 是合理的, 但是对 `job`(一次性作业) 很危险. 工业产品应该明确区分 `service`(常驻服务), `worker`(工作任务), `job`(一次性作业), `sidecar`(辅助任务), `supervisor`(嵌套监督器), 并为每类任务设置不同默认策略."

## Dependency Note(依赖说明)

本切片依赖 `specs/005-1-failure-policy-reliability/spec.md` 里定义的失败处理流水线. **衔接条款**: 每个 **`WorkRole`(工作任务角色)** 须在 **`evaluate budget`(评估预算)** 阶段接上与本角色匹配的 **`restart limit`(重启次数限制)** 与 **`escalation policy`(升级策略)** 字段用法, 以及升级处置结果进入事件与执行路径的写法; 以上须与 **`005-1`** 正文契约一致. 本切片可在 **`005-1`** 未完成前先起草; 合并验收时, 角色默认必须能够改写流水线里 **`decide action`(决定动作)** 与 **`execute action`(执行动作)** 的最终结论.

## User Scenarios & Testing(用户场景和测试) *(mandatory(必填))*

### User Story 1(用户故事一) - 按角色套用安全默认 (Priority(优先级): P1)

作为编写业务拓扑的应用作者, 我希望在声明 `service`(常驻服务), `worker`(工作任务), `job`(一次性作业), `sidecar`(辅助任务), `supervisor`(嵌套监督器) 中任一角色的受监督单元时, 系统自动套用与该角色职责相匹配的默认监督行为, 以免将适用于常驻进程的 **`Permanent`(永久重启)** 策略误用于本应在单次运行结束后收敛的一次性作业.

**Why this priority(为什么是这个优先级)**: 角色错配会导致重复执行同类工作, 资金或状态重复提交, 以及与业务认定的成功不一致的循环往复.

**Independent Test(独立测试)**: 为每个角色准备一份最小的示例拓扑, 从外部检查在只用默认策略且不额外覆写时, 成功退出与失败退出触发的自动动作是否与 **`published behavior table`(对外公开的一份行为对照表)** 一致.

**Acceptance Scenarios(验收场景)**:

1. **Given(假设)** 某个受监督单元声明为 `job`(一次性作业) 并使用角色默认策略, **When(当)** 进程以契约定义的成功退出条件结束, **Then(则)** 监督器不得为其自动启动新一轮执行, 除非调用方明确说要再来一轮任务.
2. **Given(假设)** 某个受监督单元声明为 `service`(常驻服务) 并使用角色默认策略, **When(当)** 进程以成功退出结束, **Then(则)** 默认策略允许将该 `service`(常驻服务) 单元视为仍需保持在线而有节制地自动再起, 直到被显式停止或达到保护预算.
3. **Given(假设)** 某个受监督单元声明为 `worker`(工作任务), **When(当)** 任务失败, **Then(则)** 默认策略须与 **`published behavior table`(对外公开的一份行为对照表)** 中 `worker`(工作任务) 行的条目一致: 限次数重试并拉长重试间隔, 不得按常驻模式无上限反复拉起, 并在达到角色默认预算后进入可观察的停机或升级路径.
4. **Given(假设)** 某个受监督单元声明为 `sidecar`(辅助任务), **When(当)** 其所附属的主 `service`(常驻服务) 仍满足契约给出的就绪判定而 `sidecar`(辅助任务) 失败, **Then(则)** 默认策略须允许单独重启辅助进程且默认不连带关掉主进程, 除非配置显式绑定生命周期.
5. **Given(假设)** 某个受监督单元声明为 `supervisor`(嵌套监督器), **When(当)** 子监督树报告内部失败, **Then(则)** 外层默认策略必须把内层监督树作为单一单元核算 **`restart limit`(重启次数限制)** 与 **`MeltdownTracker`(熔断跟踪器)** 口径下的预算, 避免出现外层与内层各有一套计数却无法在同一 **`TypedSupervisionEvent`(类型化监督事件)** 链条中对账的情形.

---

### Edge Cases(边界情况)

- 角色缺失或未知时, 系统必须回落到 **`documented conservative default`(文档写明的一档保守兜底默认)** , 并在诊断中标注已启用安全回退, 不得在无任何提示的情况下默认启用 **`Permanent`(永久重启)**.
- 运行中热更新角色声明时, 必须定义是否仅影响下一轮执行; 若本版本不支持热更新, 必须明确报错或拒绝并提示需要重建监督单元.
- 当用户显式覆写默认且覆写与角色语义不一致时, 系统必须输出醒目的警告或拒绝加载, 具体严格度在计划中二选一并写清.

## Requirements(需求) *(mandatory(必填))*

### Functional Requirements(功能需求)

- **FR-001**: 系统必须为 `service`(常驻服务), `worker`(工作任务), `job`(一次性作业), `sidecar`(辅助任务), `supervisor`(嵌套监督器) 五类 **`work role`(工作任务角色)** 提供对外完整且可逐项核对的默认监督行为说明, 至少覆盖成功退出, 失败退出, 人为停止, 超时与预算耗尽五类情形下系统自动采取的监督动作; 其中 `job`(一次性作业) 在成功退出后默认不得按 **`Permanent`(永久重启)** 的含义再来一轮, `service`(常驻服务) 默认允许在成功退出后为了保持在线而再拉起, 其余三类角色的默认必须分别写明: `worker`(工作任务) 限次数重试并在用尽预算后停下或升级; `sidecar`(辅助任务) 可以单独重启辅助进程且不连带关掉主进程除非配置把生命周期绑在一起; `supervisor`(嵌套监督器) 外层把整个内层监督树当一个单元来算重启与预算.

### Key Entities(关键实体) *(include if feature involves data(涉及数据时填写))*

- **`WorkRole`(工作任务角色)**: 受监督单元的分类标签, 决定绑定哪一个 **`RoleDefaultPolicyPack`(角色默认策略包)** 以及随文档发布的角色说明文案.
- **`RoleDefaultPolicyPack`(角色默认策略包)**: 绑定到某一 `WorkRole`(工作任务角色) 的一套成功, 失败与保护参数组合, 可被用户策略显式替换但不可被静默混用两个角色包.
- **`SuccessExitSemantics`(成功退出语义)**: 写明哪种退出算业务上的成功, 用来阻止对 `job`(一次性作业) 的多余重启.

## Constitution Alignment(宪章一致) *(mandatory(必填))*

### Supervision Contract(监督契约)

- **Lifecycle impact(生命周期影响)**: 引入角色维度后, 成功退出后是否自动再起取决于角色默认; **此为对外可见监督行为的显著变更**, 须在迁移文档中单设章节说明.
- **Failure behavior(失败行为)**: 失败仍走同一套失败流水线, 但 **`decide action`(决定动作)** 必须能够读取合并角色默认与用户覆写之后的最终生效策略.
- **Shutdown behavior(关闭行为)**: 用户显式关掉或取消仍然最优先, 角色默认不得覆盖这一点, 除非配置写明主辅或服务连锁停启.

### Rust Boundary and Observability Requirements(Rust 边界和可观察性需求)

- **Module ownership(模块所有权)**: 角色解析与默认策略打包必须落在配置与监督规范模块里, 不得在运行时分散硬编码多套默认.
- **Compatibility exports(兼容导出)**: None(无)
- **Diagnostics(诊断)**: 事件与错误信息须载明可唯一识别的 **`WorkRole`(工作任务角色)** 以及是否启用了兜底默认, 以便排查配置是否与意图一致.
- **Dependency impact(依赖影响)**: N/A(不适用) 除非计划阶段证明需要引入新的配置写法带来的依赖.

### Chinese Writing(中文写作)

- **Writing language(写作语言)**: 本文档必须使用中文.
- **Term format(术语格式)**: 英文术语必须写成 `English(中文说明)`.
- **Forbidden style(禁止风格)**: 禁止非中文写作, 片段式语言, 生僻词和方言.

## Success Criteria(成功标准) *(mandatory(必填))*

### Measurable Outcomes(可衡量结果)

- **SC-001**: 在五个角色的标准验收样例上, 成功退出场景中 `job`(一次性作业) 用例的自动再起比例为 0%, `service`(常驻服务) 用例在契约定义的成功退出下仍保持可用的自动恢复行为与规格一致; 其余三角色按对外公开的一份行为对照表验收, 抽样一致率 100%.
- **SC-002**: 对外文档必须为每个 **`WorkRole`(工作任务角色)** 提供至少一组最小示例拓扑与对外公开的一份行为对照表, 审查者在一轮核对里能仅凭文档判定任意给定示例归属哪一角色且无互相矛盾的默认条目.
- **SC-003**: 对于显式冲突覆写, 100% 案例必须在拒绝加载或警告正文里写明可读原因, 仅凭该正文即可理解冲突点, 不允许继续运行却行为与声明角色明显矛盾.

## Assumptions(假设)

- 五类角色足以覆盖当前产品目标行业的主干场景; 更细分的行业角色通过显式策略覆写完成, 不在本切片新增第六类默认角色.
- `sidecar`(辅助任务) 与主服务的绑定关系可通过已存在或同时规划的配置引用表达; 未绑定时默认允许辅助进程单独重启而不牵动主进程.
- `supervisor`(嵌套监督器) 作为受监督单元时, 外层观察到的成功退出定义与内层根进程一致, 细节在内联 quickstart(快速开始) 中图示.
- 若配置描述里有多颗候选主服务节点, **`sidecar`(辅助任务)** 必须在配置中显式声明所附属的 **`primary`(主实例)** 标识, 否则验收场景中的"主服务仍健康"视为未定义并应在加载阶段拒绝.
